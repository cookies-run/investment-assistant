# 投资标的监控系统 — 需求摘要

> 原始文档：投资标的监控系统_明细需求文档.docx  
> 版本：V1.0 | 日期：2026年5月15日 | 状态：初稿

---

## 一、系统目标

构建自动化投资标的监控系统，覆盖 **A 股基金（场外开放式基金）** 和 **沪深 A 股个股** 两大类，实现：

- **定时/实时涨跌监控**
- **自动同步基金持仓仓位**
- **根据止盈止损策略自动预警提醒**

---

## 二、核心功能模块

| 模块 | 核心职责 |
|---|---|
| **数据采集层** | 对接行情数据源，获取基金净值、个股实时价格、基金持仓等信息 |
| **基金监控模块** | 定时采集基金数据，同步仓位，根据持仓个股涨跌加权预估基金当日涨跌幅 |
| **个股监控模块** | 交易时间内高频采集个股价格，基于买入价实时计算盈亏 |
| **策略引擎** | 执行单日波动 / 多日累计波动两类策略判断，触发预警 |
| **通知模块** | 通过微信、邮件、短信等渠道推送预警通知 |
| **用户配置模块** | 管理监控标的、策略参数、通知偏好等 |

---

## 三、监控策略

### 3.1 触发逻辑一：单日涨跌幅触发

当单个交易日内涨跌幅绝对值超过设定阈值时，立即触发预警。

- **触发止盈**：当日涨跌幅 ≥ 单日止盈线
- **触发止损**：当日涨跌幅 ≤ -单日止损线
- **适用标的**：基金、个股
- **触发时机**：基金在 14:50 计算后判断；个股在每次高频采集后实时判断

### 3.2 触发逻辑二：多日累计波动触发

当单日波动未达阈值，但连续多个交易日累计涨跌幅达到阈值时，触发预警。

- **多日累计涨跌幅** = Σ 近 N 个交易日每日涨跌幅
- **触发止盈**：多日累计涨跌幅 ≥ 多日累计止盈线
- **触发止损**：多日累计涨跌幅 ≤ -多日累计止损线
- **适用标的**：基金、个股
- **触发时机**：每日收盘后计算判断

### 3.3 默认策略参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| 单日止盈线 | 3% | 单日涨幅达到触发止盈提醒 |
| 单日止损线 | 3% | 单日跌幅达到触发止损提醒 |
| 累计止盈线 | 8% | 多日累计涨幅达到触发止盈提醒 |
| 累计止损线 | 8% | 多日累计跌幅达到触发止损提醒 |
| 累计天数窗口 | 5 天 | 计算累计涨跌幅的交易日窗口长度 |
| 个股监控频率 | 1 分钟 | 交易时间内高频采集间隔 |

## 四、监控调度时间线

### 4.1 基金监控

| 时间 | 执行动作 | 说明 |
|---|---|---|
| 09:30 | 开盘前准备 | 检查持仓数据是否为最新，确认当日为交易日 |
| **14:50** | **涨跌预估计算** | 采集持仓个股实时行情，加权计算基金预估涨跌幅，执行策略判断 |
| 15:00 | 收盘数据确认 | 等待个股收盘数据稳定，更新当日涨跌幅记录 |
| 15:30 | 仓位数据同步 | 尝试同步基金最新持仓信息（如有更新） |
| 18:00 | 净值校验 | 采集基金当日净值，与预估值对比，记录偏差 |

### 4.2 个股监控

| 时间 | 执行动作 | 说明 |
|---|---|---|
| 09:25 | 集合竞价观察 | 采集集合竞价数据，判断开盘价是否触及止盈止损线 |
| 09:30-11:30 | 上午盘高频监控 | 每 1 分钟采集一次行情，实时执行策略判断 |
| 13:00-15:00 | 下午盘高频监控 | 每 1 分钟采集一次行情，实时执行策略判断 |
| 15:00 | 收盘处理 | 记录当日收盘价和涨跌幅，更新多日累计值 |
| 15:05 | 多日累计判断 | 基于收盘数据执行多日累计波动策略判断 |

> **非交易日**：自动识别周末、法定节假日，暂停所有监控任务。

## 五、核心数据模型

| 表名 | 说明 |
|---|---|
| **Fund** | 基金信息及策略参数配置 |
| **Stock** | 个股信息、买入价格及策略参数配置 |
| **Fund_Holding** | 基金持仓股票列表（股票代码、名称、持仓占比、报告日期） |
| **Daily_Record** | 每日涨跌记录（开盘价、收盘价、涨跌幅、多日累计涨跌幅） |
| **Alert_Record** | 预警记录（触发类型、触发值、阈值、通知状态、触发时间） |

## 六、非功能性需求

| 维度 | 要求 |
|---|---|
| **可靠性** | 采集失败自动重试（最多3次）；连续失败超阈值发送异常通知；涨跌幅计算保留至少2位小数；所有记录持久化存储 |
| **实时性** | 个股从数据更新到策略判断延迟不超过监控间隔的50%；预警触发到通知发出延迟不超过1分钟 |
| **可扩展性** | 支持后续扩展债券、ETF、期货等标的类型；策略引擎采用插件化设计 |
| **安全性** | 持仓信息、买入价格等敏感数据加密存储；对外 API 进行身份认证和访问控制 |
| **可维护性** | 完善的日志记录（数据采集、策略判断、通知发送） |

## 七、待确认事项（遗留问题）

| 序号 | 待确认事项 | 影响范围 |
|---|---|---|
| 1 | **行情数据源选择** | 影响数据采集模块的开发方式和成本（免费/付费接口） |
| 2 | **基金持仓数据同步频率** | 基金持仓通常按季度披露，需确认用户是否接受数据滞后 |
| 3 | **通知渠道优先级** | 确定首选通知方式（微信/邮件/短信），影响通知模块开发 |
| 4 | **个股监控频率下限** | 高频监控对系统资源和数据源并发有要求，需确认可接受范围 |
| 5 | **是否需要用户界面** | 影响前端开发工作量，确认是纯后端服务还是配套管理界面 |
| 6 | **部署环境** | 本地部署 / 云服务器 / Serverless，影响整体架构设计 |
| 7 | **是否需要历史数据回测** | 确认是否要对历史行情数据进行策略回测验证 |
| 8 | **多日累计波动的计算方式** | 确认用户期望使用简单累加还是复合计算 |

---

## 八、后续建议

1. **优先确认待确认事项 1、5、6**（数据源、是否需前端、部署环境），这三项直接决定技术选型和架构方向。
2. **明确基金持仓数据来源**（天天基金、蚂蚁基金、季报披露等），影响同步模块的实现。
3. **若需前端界面**，可进一步输出 UI 原型或页面流程设计。
